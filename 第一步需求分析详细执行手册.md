### 二、Token限制与上下文管理策略

#### 2.1 核心概念

```
┌─────────────────────────────────────────────────────────┐
│                    上下文窗口（Context Window）            │
│                                                         │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌────────┐  │
│  │ System   │  │ 历史消息  │  │ 当前输入  │  │ AI输出 │  │
│  │ Prompt   │  │ (多轮)   │  │          │  │        │  │
│  │ ~2K token│  │ 可变长度  │  │ 可变长度  │  │ 可变   │  │
│  └──────────┘  └──────────┘  └──────────┘  └────────┘  │
│                                                         │
│  ← ───── 所有部分的token总和不能超过上下文窗口 ───── →     │
└─────────────────────────────────────────────────────────┘
```

**关键数字（用于估算）：**
- 1个中文字 ≈ 0.6个token
- 1个英文单词 ≈ 1.3个token
- 1行Java代码 ≈ 10-15个token
- RuoYi的BaseEntity源码 ≈ 500 token
- RuoYi的SysUserController源码（357行）≈ 4000 token
- 一个完整的需求文档（5页A4）≈ 3000-5000 token
- 一个完整的结构化需求JSON ≈ 2000-4000 token

#### 2.2 上下文分配策略

针对本流程，建议按以下方式分配上下文窗口（以128K为例）：

| 组成部分 | 预算token | 占比 | 说明 |
|---------|----------|------|------|
| System Prompt（RuoYi规范） | ~3,000 | 2.3% | 框架约束、编码红线、技术栈 |
| 框架源码上下文 | ~5,000 | 3.9% | BaseEntity/BaseController等核心类 |
| 当前任务指令 | ~1,000 | 0.8% | 本次具体要AI做什么 |
| 需求文档内容 | ~5,000 | 3.9% | 业务需求原文 |
| 历史对话记录 | ~10,000 | 7.8% | 多轮对话中的上下文 |
| **留给AI输出** | **~30,000** | **23.4%** | 生成代码/文档的空间 |
| **安全余量** | **~74,000** | **57.8%** | 防止溢出 |

> **关键原则：永远给AI输出预留足够空间。** 如果输入占了90%的上下文，AI的输出会被截断。

#### 2.3 长文档分段处理策略

当需求文档超长（>50页），或需要分析大量源码时，使用以下策略：

**策略一：Map-Reduce（分段摘要合并）**
```
原始文档（100页）
    ↓ 切分
[段1] [段2] [段3] [段4] [段5]
    ↓ 每段独立让AI提取要点
[摘要1] [摘要2] [摘要3] [摘要4] [摘要5]
    ↓ 合并所有摘要，再让AI做最终分析
[最终完整性报告]
```

**策略二：Refine（迭代精炼）**
```
[段1] → AI生成初步分析
    ↓ 把初步分析 + [段2] → AI更新分析
    ↓ 把更新分析 + [段3] → AI继续更新
    ↓ ...
[最终分析]（逐步精炼，保持上下文连贯）
```

**策略三：RAG（检索增强生成） — 推荐用于源码分析**
```
RuoYi源码 → 切分成代码片段 → 向量化存储
    ↓
用户提问："BaseEntity有哪些字段？"
    ↓ 向量检索最相关的代码片段
检索到 BaseEntity.java 的片段
    ↓ 把片段 + 问题一起发给AI
AI回答（带有准确的源码引用）
```

**LangChain4j 为我们解决的问题（不需要自己写）：**

| 能力 | LangChain4j提供的方案 | 不用LangChain4j要自己写什么 |
|------|----------------------|--------------------------|
| 调用多种大模型API | 统一接口，一行代码切换模型 | 每个API写一套HTTP调用代码 |
| 上下文记忆管理 | `TokenWindowChatMemory`自动管理 | 自己实现token计数和消息裁剪 |
| 长文档切分 | `DocumentSplitter`系列（按段落/句子/字符/正则/递归） | 自己写文本切分逻辑 |
| 文档解析 | Apache Tika集成（支持Word/PDF/Markdown/HTML） | 自己集成POI+PDFBox等 |
| RAG检索增强 | 内置向量存储+检索+注入 | 自己实现向量数据库对接 |
| Token计数 | `TokenCountEstimator` | 自己实现tiktoken估算 |
| 结构化输出 | AI Service自动解析JSON | 自己写JSON解析和校验 |

#### 3.2 上下文记忆管理 — LangChain4j内置

```java
// 方式一：按消息条数管理（简单，适合原型）
ChatMemory memory = MessageWindowChatMemory.builder()
    .maxMessages(20)  // 保留最近20条消息
    .build();

// 方式二：按token数管理（推荐生产使用）
ChatMemory memory = TokenWindowChatMemory.builder()
    .maxTokens(8000)  // 保留最近8000个token的消息
    .tokenCountEstimator(new OpenAiTokenCountEstimator("gpt-4o"))
    .build();
// 超出限制时自动淘汰最老的消息，保证不超过模型上下文窗口
```

#### 3.3 文档解析与切分 — LangChain4j内置

```java
// 1. 解析文档（Word/PDF/Markdown/HTML通吃）
DocumentParser parser = new ApacheTikaDocumentParser();
Document doc = FileSystemDocumentLoader.loadDocument("需求文档.docx", parser);

// 2. 切分成适合AI处理的片段
DocumentSplitter splitter = DocumentSplitters.recursive(
    1000,   // 每个片段最大1000个token
    200     // 片段之间重叠200个token（保持上下文连贯）
);
List<TextSegment> segments = splitter.split(doc);

// 3. 逐段发给AI处理，或存入向量数据库做RAG
```

**内置的切分策略：**

| 切分器 | 策略 | 适用场景 |
|--------|------|---------|
| `DocumentByParagraphSplitter` | 按段落切分 | 结构清晰的文档 |
| `DocumentBySentenceSplitter` | 按句子切分 | 需要细粒度的场景 |
| `DocumentByCharacterSplitter` | 按字符数切分 | 最简单粗暴 |
| `DocumentSplitters.recursive()` | **递归切分（推荐）** | 先按段落，段落太大按句子，句子太大按字符 |

### 做什么

### 怎么做

**方式一：已有文档（手动方式 — 适合用Windsurf/ChatGPT等对话式AI）**

- 支持格式：`.docx`、`.pdf`、`.md`、`.txt`
- **注意token限制：** 如果文档超过20页（约15,000 token），需要分段喂给AI（见上方"长文档分段处理策略"）

**方式一B：已有文档（程序化方式 — 适合集成到RuoYi后台）**
- 使用LangChain4j + Apache Tika自动解析文档：

```java
// 后端Service代码示例
DocumentParser parser = new ApacheTikaDocumentParser();
Document doc = FileSystemDocumentLoader.loadDocument(uploadedFilePath, parser);

// 估算token数，决定是否需要切分
int estimatedTokens = tokenCountEstimator.estimateTokenCount(doc.text());
if (estimatedTokens > 10000) {
    // 文档太长，需要切分后分段处理
    DocumentSplitter splitter = DocumentSplitters.recursive(8000, 500);
    List<TextSegment> segments = splitter.split(doc);
    // 使用Map-Reduce策略逐段分析后合并
} else {
    // 文档长度OK，直接一次性发给AI
    String response = chatModel.generate(systemPrompt + doc.text());
}
```

## 1.3 AI解析文档提取关键信息

### 做什么

把需求文档交给AI，让AI从中提取5类关键信息。

### Token预算

本步骤的token消耗估算：
- System Prompt（RuoYi规范）：~2,000 token
- 任务指令：~500 token
- 需求文档内容：~3,000-10,000 token（视文档长度）

- **总计：~8,500-17,500 token**（128K模型绰绰有余，无需切分）

> 如果需求文档超过30,000 token（约50页），请先用Map-Reduce策略切分处理（见上方2.3节）。

## 1.7 AI生成结构化需求JSON

### 做什么

把需求转成机器可读的JSON格式，这是后续所有步骤的源头数据。

### Token预算

本步骤的token消耗估算：
- System Prompt（复用1.3的）：~2,000 token
- 前面步骤的对话历史（如果是多轮对话）：~5,000-10,000 token
- 任务指令 + Schema定义：~2,000 token
- **预期AI输出（完整JSON）：~3,000-6,000 token**（2个实体约4,000 token）
- **总计：~12,000-20,000 token**

> **重要：** 如果实体数量很多（>5个表），JSON会很长。建议分批生成——每次让AI生成2-3个实体的JSON，最后手动合并。这样每次调用都在token安全范围内。

**分批生成策略（>5个实体时）：**
```
第1次调用：生成实体1-3的fields + 对应的dictionaries
第2次调用：生成实体4-6的fields + 对应的dictionaries
第3次调用：生成menus和dataPermissions（需要知道所有实体名）
最后：手动合并为完整JSON
```

## 1.8 人工审查并补充

### 做什么

### 审查清单

**① 检查功能点是否遗漏**
- [ ] AI提取的功能点与你心中的需求一致？
- [ ] 是否有AI没有理解到的隐含需求？

**② 检查字段设计**
- [ ] 每个字段的类型是否正确？（如手机号应该是varchar(11)而不是int）
- [ ] 必填字段标注是否合理？
- [ ] 查询条件标注是否合理？（用户会通过哪些字段搜索？）
- [ ] 字典字段是否都用了字典？（状态/类型/级别等字段不应该直接写中文）

**③ 检查关联关系**
- [ ] 实体之间的关系是否正确？（一对多/多对多）
- [ ] 级联规则是否合理？（删除主表时子表怎么处理？）
- [ ] 外键字段类型是否与被关联表的主键类型一致？

**④ 检查RuoYi适配**
- [ ] AI标注的"复用"是否真的能复用？
- [ ] AI标注的"新建"是否真的需要新建？
- [ ] 数据权限方案是否满足你的需求？

**⑤ 检查结构化JSON**
- [ ] JSON中的字段与你确认的字段表一致？
- [ ] 字典数据完整？每个枚举值都有？
- [ ] 菜单权限标识命名合理？

## 附录B：多轮对话 vs 独立调用的选择

| 方式 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| **多轮对话**（1.3→1.4→1.5在同一会话中） | AI能记住前面的分析结果，输出更连贯 | 历史消息累积越来越多，后面的调用token更贵 | 对话式AI（Windsurf/ChatGPT） |
| **独立调用**（每步独立，把上一步结果作为输入） | 每次调用token可控，成本可预测 | 需要手动传递上一步结果 | 程序化集成（RuoYi后台AI功能） |

**推荐做法：**
- 手动操作（用Windsurf/ChatGPT）→ **多轮对话**，简单直接
- 程序化集成（RuoYi后台）→ **独立调用** + `TokenWindowChatMemory`自动管理

**多轮对话中的token管理（LangChain4j自动处理）：**
```java
// TokenWindowChatMemory 会自动：
// 1. 每次对话后把消息存入记忆
// 2. 当总token超过maxTokens时，自动淘汰最老的消息
// 3. System Prompt永远保留（不会被淘汰）
// → 你只需要关注业务逻辑，不需要手动管理token

ChatMemory memory = TokenWindowChatMemory.builder()
    .maxTokens(60000)  // 128K模型中分配60K给记忆
    .tokenCountEstimator(new OpenAiTokenCountEstimator("gpt-4o"))
    .build();
```

## 附录C：常见问题

### Q1：AI输出被截断了怎么办？

**原因：** 单次输出超过了`maxTokens`限制（通常默认4096）。

**解决：**
- 调大`maxTokens`参数（DeepSeek最大支持8192，GPT-4o最大支持16384）
- 或者让AI分段输出："请先输出实体1-3的字段定义，我会再让你输出剩余的"

### Q2：AI生成的JSON格式不对怎么办？

**解决方案一（手动）：** 告诉AI"请严格按照JSON格式输出，不要有多余的说明文字"

**解决方案二（LangChain4j结构化输出）：**
```java
// 定义Java类，LangChain4j自动把AI输出解析为Java对象
@Description("结构化需求")
record RequirementSpec(
    String module,
    String moduleName,
    List<EntitySpec> entities,
    List<DictSpec> dictionaries
) {}

interface RequirementGenerator {
    @UserMessage("根据需求生成结构化JSON：{{requirement}}")
    RequirementSpec generate(@V("requirement") String requirement);
}
// LangChain4j会自动在Prompt中注入JSON Schema，确保AI输出严格的JSON格式
```

### Q3：不同步骤可以用不同模型吗？

**可以，也推荐这样做：**
- 1.3（文档提取）→ DeepSeek V3（便宜，够用）
- 1.4（适配分析）→ DeepSeek V3
- 1.5（完整性评估）→ DeepSeek V3
- 1.7（JSON生成）→ DeepSeek R1 或 GPT-4o（需要更精确的结构化输出）

```java
// LangChain4j支持给不同的AI Service配不同模型
RequirementAnalyzer lightAnalyzer = AiServices.builder(RequirementAnalyzer.class)
    .chatLanguageModel(deepseekV3)  // 便宜的模型做分析
    .build();

JsonGenerator preciseGenerator = AiServices.builder(JsonGenerator.class)
    .chatLanguageModel(deepseekR1)  // 精确的模型做JSON生成
    .build();
```

### Q4：担心API Key泄露怎么办？

参见上方"五、API Key安全管理"。核心原则：
1. **不要**把Key写在代码里
2. 用环境变量或RuoYi的`sys_config`表存储
3. `.gitignore`中排除包含Key的配置文件

### Q5：没有外网，能离线使用吗？

**可以，使用Ollama本地运行模型：**
```java
// 安装Ollama后，拉取模型：ollama pull qwen2.5:14b
// LangChain4j对接Ollama：
ChatLanguageModel localModel = OllamaChatModel.builder()
    .baseUrl("http://localhost:11434")
    .modelName("qwen2.5:14b")        // 本地14B参数模型
    .build();
// 完全离线，无API费用，但需要16GB+显存
```

需要额外Maven依赖：
```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-ollama</artifactId>
    <version>1.12.2</version>
</dependency>
```
